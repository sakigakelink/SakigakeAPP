# Sakigake Shift 機能要件書

**文書番号**: SS-FRD-001
**版数**: 1.0
**作成日**: 2026年3月20日
**製品名**: Sakigake Shift（Sakigake Prime オプションモジュール）

---

## 1. 製品概要

### 1.1 目的
Sakigake Shiftは、病棟看護師のシフト（勤務表）を数理最適化技術により自動生成するシステムである。Google OR-Tools CP-SATソルバーを搭載し、スタッフの希望・夜勤公平性・労働基準法を同時に満たす最適なシフトを複数案提示する。

### 1.2 対象ユーザー
病棟師長、看護管理者、勤務表作成担当者

### 1.3 提供形態
クラウド型AIネイティブ電子カルテ「Sakigake Prime」のオプション機能として提供する。Webブラウザから利用可能。

---

## 2. スタッフ管理

### 2.1 勤務体系
システムは以下の勤務体系を管理する。

| 勤務体系 | 説明 | 対象シフト |
|----------|------|-----------|
| 2交代制 | 日勤・夜勤（night2）・明け（ake） | day, late, night2, ake, off, paid, refresh |
| 3交代制 | 日勤・準夜勤・深夜勤 | day, late, junnya, shinya, off, paid, refresh |
| 日勤専従 | 日勤のみ | day, late, off, paid, refresh |
| 夜勤専従 | 夜勤のみ | night2/ake または junnya/shinya, off, paid, refresh |
| 固定勤務 | 曜日ごとの固定パターン | 固定パターンに従う |

### 2.2 職種区分
- **看護師（nurse）**: 全シフトに配置可能
- **准看護師（junkango）**: 夜勤は1シフトあたり最大1名の制約あり
- **看護助手（nurseaide）**: 日勤系のみ配置可能

### 2.3 個人パラメータ
各スタッフに以下を設定可能とする。

- **maxNight**: 月間夜勤上限回数（ハード制約）
- **minNight**: 月間夜勤下限回数（ソフト制約、未指定時は自動算出）
- **nightRestriction**: 準夜専門／深夜専門の制限（3交代制のみ）
- **fixedPattern**: 固定勤務の曜日別パターン

## 3. 病棟設定

### 3.1 人員配置要件

| 設定項目 | 制約種別 | 説明 |
|----------|----------|------|
| reqDayWeekday | ソフト | 平日の日勤必要人数 |
| reqDayHoliday | ソフト | 休日の日勤必要人数 |
| reqJunnya | **ハード** | 準夜勤の1日あたり必要人数（厳密一致） |
| reqShinya | **ハード** | 深夜勤の1日あたり必要人数（厳密一致） |
| reqLate | ソフト | 遅出の1日あたり必要人数 |
| monthlyOff | **ハード** | 月間公休日数（厳密一致） |

### 3.2 曜日別・日付別オーバーライド
- **dayStaffByDay**: 曜日ごとの日勤人数指定（月〜日）
- **dayDateOverrides**: 特定日付の人数・有資格者数指定
- **minQualifiedByDay**: 曜日ごとの有資格者最低人数

---

## 4. 希望シフト

### 4.1 希望の種類
- **assign（希望勤務）**: 指定日に指定シフトを割り当てる
- **avoid（希望回避）**: 指定日に指定シフトを割り当てない

### 4.2 制約としての扱い
**全ての希望はハード制約として処理する。緩和・優先度付けは行わない。** スタッフが入力した希望は100%反映される。ソルバーが希望を満たせない場合は、希望を削るのではなく解なし（infeasible）を返す。

---

## 5. ソルバー（制約最適化エンジン）

### 5.1 最適化エンジン
Google OR-Tools CP-SAT（Constraint Programming with SAT）ソルバーを使用する。全CPUコアによる並列探索を行う。

### 5.2 ハード制約一覧
以下の制約は必ず満たされる。違反する解は生成されない。

| # | 制約名 | 内容 |
|---|--------|------|
| H1 | 準夜勤人数 | 毎日reqJunnya名を厳密配置 |
| H2 | 深夜勤人数 | 毎日reqShinya名を厳密配置 |
| H3 | 月間公休日数 | 全スタッフがmonthlyOff日の公休を取得 |
| H4 | 希望シフト反映 | assign/avoidを全件反映 |
| H5 | 7連勤禁止 | いかなる8日間にも必ず1日以上の休日を含む |
| H6 | 夜勤上限 | 各スタッフのmaxNightを超えない |
| H7 | 2交代：night2→ake連動 | night2の翌日は必ずake |
| H8 | 2交代：ake後休息 | akeの翌日はoff/paid/refreshのいずれか |
| H9 | 2交代：ake前提条件 | akeの前日にnight2がなければならない |
| H10 | 3交代：準夜後制限 | 準夜勤の翌日は準夜/off/paid/refreshのみ |
| H11 | 3交代：3連続夜勤禁止 | 準夜・深夜を問わず3日連続の夜勤を禁止 |
| H12 | 3交代：準夜→深夜禁止 | 準夜勤の翌日に深夜勤を割り当てない |
| H13 | 3交代：遅出→深夜禁止 | 遅出の翌日に深夜勤を割り当てない |
| H14 | 准看護師夜勤制限 | 1シフトあたり准看護師は最大1名 |
| H15 | 勤務体系制限 | 各勤務体系で許可されないシフトは割り当てない |
| H16 | 前月引継ぎ | 前月末日の勤務状態を考慮し、連勤・夜勤パターンの連続性を保証 |

### 5.3 ソフト制約一覧
以下の制約はペナルティとして最適化される。可能な限り満たすが、ハード制約との兼ね合いで違反を許容する。

| # | 制約名 | ペナルティ重み | 内容 |
|---|--------|---------------|------|
| S1 | 日勤人数充足 | 500pt/人/日 | reqDayWeekday/Holidayへの不足 |
| S2 | 夜勤下限 | 300pt/枠 | minNight未達の場合 |
| S3 | 5連勤抑制 | 500pt/検出 | 5日連続勤務パターン |
| S4 | 6連勤抑制 | 300pt/検出 | 6日連続勤務パターン |
| S5 | 夜勤間隔 | 15〜30pt | 夜勤間が2〜3日と近い場合 |
| S6 | 深夜後休息 | 3〜20pt | 深夜勤の翌日に休息がない場合（3交代） |
| S7 | 散発夜勤 | 3〜15pt | 深夜→休→深夜パターン（3交代） |
| S8 | 準夜→休→深夜 | 50pt | 体内リズム破壊パターン（3交代） |
| S9 | 日勤→深夜 | 25pt | 勤務間インターバルが短い遷移（3交代） |
| S10 | 良好ローテーション | -15pt（ボーナス） | 深夜→休→準夜など理想的パターン |
| S11 | 遅出均等化 | 20pt×差 | 遅出回数の最大-最小差を縮小 |
| S12 | 週末均等化 | 40pt×差 | 週末勤務回数の最大-最小差を縮小 |
| S13 | 準夜深夜バランス | 10〜15pt | 準夜と深夜の回数差を縮小（3交代） |
| S14 | 夜勤公平性（Gini） | 200〜1500pt（段階式） | 夜勤回数差1回=200pt、5回以上=1500pt |

### 5.4 ソフト制約のペナルティ適応
S5〜S10のペナルティ重みは、スタッフのrefresh（リフレッシュ休暇）取得回数に応じて自動的に低減される。リフレッシュ休暇が3回以上のスタッフは基本重みの20%程度に軽減される。

### 5.5 実行モード

#### 5.5.1 標準モード
シード値（7, 31, 97）を順次試行し、各15秒のタイムアウトで探索する。全シードがタイムアウトした場合、最良シードを45秒に延長して再探索する。

#### 5.5.2 複数案生成モード（Pool）
5つの異なる重み戦略で同時に解を生成し、多角的なシフト案を提示する。

| 戦略 | 重点 |
|------|------|
| balanced | 全指標を均等に重視 |
| night_fairness | 夜勤公平性を最重視 |
| consec_suppress | 連勤抑制を最重視 |
| rotation_quality | ローテーション品質を最重視 |
| staff_coverage | 人員充足を最重視 |

#### 5.5.3 緩和モード
通常モードで解が見つからない場合の診断用。夜勤下限→夜勤要件の順に制約を段階的に緩和し、実行不能の原因を特定する。

### 5.6 夜勤下限の自動算出
minNightが未指定の場合、以下のロジックで自動算出する。

1. 総夜勤需要 = (reqJunnya + reqShinya) × 月日数
2. 供給比率に応じた充足率（70〜85%）を決定
3. 各スタッフのmaxNight比率で按分

### 5.7 事前検証（プリフライトチェック）
ソルバー起動前に以下を検証し、明らかに実行不能なケースを即座に判定する。

- 夜勤可能人数 × maxNight が総夜勤需要を満たすか
- 個人の希望休日数が許容範囲内か
- 固定勤務スタッフの30日分データが完備しているか

## 6. 固定勤務スタッフの処理

### 6.1 事前配置
固定勤務スタッフのシフトは、ソルバー起動前に希望シフトから確定済みシフトとして展開される。

### 6.2 需要調整
固定勤務スタッフの配置済みシフトは日勤・準夜・深夜の必要人数から差し引かれ、残りの需要に対してソルバーが非固定スタッフを配置する。

### 6.3 30日完備性検証
固定勤務スタッフは当月全日（30日または31日）のシフトデータが必要である。不完全な場合はHTTP 400エラーを返し、ソルバーを起動しない。

## 7. シフト品質評価

### 7.1 公平性指標

| 指標 | 算出方法 | 評価基準 |
|------|----------|----------|
| 夜勤Gini係数 | 夜勤回数のジニ係数 | <0.1 優秀 / <0.2 良好 / <0.3 普通 / ≥0.3 要改善 |
| 週末Gini係数 | 週末勤務回数のジニ係数 | 同上 |
| 遅出Gini係数 | 遅出回数のジニ係数 | 同上 |
| 夜勤回数レンジ | max - min | 差が小さいほど公平 |
| 週末回数レンジ | max - min | 同上 |

### 7.2 個人別品質指標

| 指標 | 内容 |
|------|------|
| 最大連勤日数 | 当月中の最長連続勤務日数 |
| 5連勤検出数 | 5日以上連続勤務の回数 |
| 夜勤間隔（近接） | 夜勤間2〜3日の回数 |
| 深夜後無休検出 | 深夜勤翌日に休息がない回数（3交代） |
| 散発夜勤検出 | 深夜→休→深夜パターン回数（3交代） |
| 良好ローテーション | 理想的な夜勤パターン回数（3交代） |

### 7.3 労働基準法準拠チェック

| チェック項目 | 根拠法令 | 重要度 | 判定基準 |
|-------------|----------|--------|----------|
| 勤務間インターバル | 労働時間等設定改善法 | warning | シフト間の休息が11時間未満 |
| 週1日休日 | 労基法第35条 | warning | 日〜土の暦週に休日がない |
| 4週4日休日 | 労基法第35条（変形） | caution | 28日スライド窓で休日4日未満 |

## 8. ドラフト管理

### 8.1 ドラフト保存
ソルバーが生成した各シフト案はドラフトとして保存される。タイムスタンプ付きで一意に識別される。

### 8.2 ドラフト選択
複数のドラフトから1つを選択し、アクティブなシフト案として表示する。

### 8.3 シフト確定
選択したドラフトを確定（confirm）することで、当月のシフトとしてロックする。

### 8.4 確定後修正
確定済みシフトに対して、個別のスタッフ・日付単位で手動修正が可能。

### 8.5 ドラフト削除
不要なドラフトを削除できる。

---

## 9. バックアップ

### 9.1 手動バックアップ
任意のタイミングでシフトデータのバックアップを作成できる。

### 9.2 自動日次バックアップ
毎日午前3時に自動バックアップを実行する。保持期間は30日間。ただし月初のバックアップは1年間保持する。

### 9.3 バックアップ復元
バックアップ一覧から選択し、シフトデータを復元できる。

---

## 10. 前月データ引継ぎ

### 10.1 引継ぎ項目
- 前月末日の勤務種別（lastDay）
- 前月末前日の勤務種別（secondLastDay）
- 月末時点の連続勤務日数（consecutiveWork）
- 準夜・深夜の連続回数

### 10.2 異動対応
スタッフの病棟間異動時、異動元の前月データを自動補完する。

---

## 11. 様式9連携

### 11.1 様式9出力
確定済みシフトを様式9（看護必要度に関する報告書式）形式でエクスポートする。シフトコードを様式9の所定コードに変換して出力する。

---

## 12. 実績管理

### 12.1 実績入力
確定済みシフトに対して、日単位で実績（実際の勤務）を記録できる。

### 12.2 月次締め
月末に実績を確定（finalize）し、アーカイブとして保存する。締め後の変更は不可。

---

## 13. 非機能要件

### 13.1 性能
- 標準モード: スタッフ23名×30日のシフト生成を45秒以内に完了すること
- 複数案モード: 5案の生成を120秒以内に完了すること

### 13.2 対応ブラウザ
- Google Chrome（最新版）
- Microsoft Edge（最新版）

### 13.3 データ保存
- クライアント側: localStorageによるデータ永続化
- サーバー側: JSON形式のファイルベースストレージ
