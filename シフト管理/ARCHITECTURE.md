## 2. ユーザーの操作フロー

```
① 職員登録          ② 希望入力          ③ シフト生成
  職員Tabで            シフトTabで          「生成開始」
  名前・区分・         希望休・             ボタンで
  夜勤上限を登録       勤務指定を入力       ソルバー実行
      │                    │                    │
      ↓                    ↓                    ↓
④ 下書き選択        ⑤ シフト確定         ⑥ 実績管理
  複数案から            選んだ案を           確定後に
  ベストを選ぶ          確定する             変更を記録
      │                    │                    │
      ↓                    ↓                    ↓
⑦ PDF出力 / 様式9 Excel出力
```

### 各ステップの詳細

**① 職員登録** (`staff.js`)
- 職員Tabで氏名・勤務区分(二交代/三交代/日勤のみ等)・職種(nurse/junkango/nurseaide)・夜勤上限を登録
- ドラッグ&ドロップで表示順を並び替え
- JSON一括インポートにも対応
- バックエンドの `shared/employees.json` に同期

**② 希望入力 = 初期配置** (`wish.js`)
- GUI: 職員選択 → 日付チップ → 指定/除外モード切替 → シフト種別ボタンで入力
- テキスト: 「内田5休」「山田全有給」「田中5×準夜」のような自然言語風のパース
- **指定（assign）**: 特定日に特定シフトを強制（ソルバーは動かさない）
- **除外（avoid）**: 特定日の特定シフトを禁止（ソルバーはその選択肢を排除）
- 希望は全てハード制約（＝初期配置）としてソルバーに渡される

**③ シフト生成** (`shift.js` → `routes.py` → `solver_subprocess.py` → `solver.py`)
- 「生成開始」ボタン → POST `/solve-stream`
- サブプロセスでソルバー実行（Flask本体をクラッシュから隔離）
- 進捗はSSE(Server-Sent Events)でリアルタイムにブラウザへ送信
- 3つのシード(7,31,97)で各15秒、最良解を採用
- 「複数案」ボタンで5案同時生成も可能

**④ 下書き選択** (`draft.js`)
- 生成された案は自動的に下書き保存される（命名: `自動_MMDD_HHMM`）
- 各案のスコア・ペナルティ内訳・夜勤差を確認して比較
- 複数案モードでは重み違い(公平性重視/快適性重視等)の案を比較

**⑤ シフト確定** (`draft.js`)
- 選択した下書きを「確定」操作でロック
- 確定前にバックアップを自動作成
- 確定後も個別セルの手動修正は可能（変更履歴に記録）

**⑥ 実績管理** (`actual.js`)
- 確定後に「実績入力開始」で実績モードに遷移
- 予定⇔実績を切替表示
- 変更には理由の入力が必要
- 「実績確定」で最終ロック（様式9出力が可能になる）

**⑦ 出力** (`export.js`)
- PDF: シフト表をPDF形式で出力
- 様式9: 病棟別に厚労省様式のExcelを生成

## 6. ソルバーの動作概要

### solver.pyの処理フロー

```
ShiftSolver.__init__()
  ├─ 職員を3グループに分離
  │   ├─ fixed_staff  : 固定シフト（ソルバー対象外）
  │   ├─ flex_staff   : flexRequest（手動入力）
  │   └─ staff_list   : ソルバー対象（2kohtai/3kohtai/day_only/night_only）
  ├─ maxNight をバックエンドから補正
  ├─ minNight を自動計算（夜勤需要÷対象者数）
  └─ 病棟別config読込

solve()
  ├─ 事前チェック（A〜E）
  │   ├─ A: 夜勤供給不足の検出
  │   ├─ B: 公休日数オーバーの検出
  │   ├─ C: 勤務区分と希望の整合性
  │   ├─ D: 前月引継ぎと希望の競合
  │   └─ E: 同日の矛盾する希望
  │
  ├─ _solve_core() を3シード(7,31,97)で実行
  │   ├─ CP-SATモデル構築
  │   │   ├─ 変数: shifts[職員][日][シフト] = 0 or 1
  │   │   ├─ 初期配置の固定（指定→==1、除外→==0、前月末制約）
  │   │   ├─ ハード制約（§3参照）
  │   │   └─ ソフト制約 → 目的関数（§4参照）
  │   ├─ ソルバー実行（各15秒）
  │   └─ 解の抽出（シフト割当 + スコア）
  │
  ├─ 最良解を選択（OPTIMAL優先、なければ最小obj）
  └─ 結果返却
```

### サブプロセス隔離の理由

OR-Tools (C++) は内部エラー時に `abort()` でプロセスごと終了することがある。Flaskプロセスのクラッシュを防ぐため、ソルバーを別プロセスで実行する。

```
Flask (安定)  ──Popen──→  solver_subprocess.py (クラッシュしても影響なし)
              ←stdout──   進捗 (JSONL)
              ←file────   結果 (JSON)
```

stdoutではなくファイル経由で結果を返す理由: OR-Toolsの C++ stderr出力がstdoutに混入するバッファリング問題を回避。

## 7. フォーマット変換

フロントエンドとバックエンドでデータ形式が異なる。`validation.py` が変換を担当。

### 勤務区分

| バックエンド (employees.json) | フロントエンド (UI/API) |
|------------------------------|------------------------|
| `shiftCategory: "twoShift"` | `workType: "2kohtai"` |
| `shiftCategory: "threeShift"` | `workType: "3kohtai"` |
| `shiftCategory: "dayOnly"` | `workType: "day_only"` |
| `shiftCategory: "nightOnly"` | `workType: "night_only"` |
| `shiftCategory: "flexRequest"` | `workType: "fixed"` |

### 病棟

| バックエンド | フロントエンド |
|-------------|--------------|
| `"ichiboutou"` | `"1"` |
| `"nibyoutou"` | `"2"` |
| `"sanbyoutou"` | `"3"` |

### 夜勤上限

| バックエンド | フロントエンド |
|-------------|--------------|
| `personalRules.nightShift.maxPerMonth: 5` | `maxNight: 5` |

### スキルポイント

| バックエンド | フロントエンド |
|-------------|--------------|
| `skillPoint: 3` | `skillPoint: 3` |

各職員に0〜5の整数でスキルレベルを設定（0=新人、5=ベテラン、デフォルト3）。
ソルバーは各時間帯のスキルポイント合計が最低値を満たすようソフト制約で最適化し、
夜勤帯ではskillPoint=0の職員が2名以上同一帯に入らないハード制約を適用する。
設定は `engines/{ward}/config.json` の `skillPointMinimumPerBand` と `noDoubleRookie` で病棟別に管理。

## 8. 安全設計

| 項目 | 実装 |
|------|------|
| **原子的書き込み** | 全JSONファイル: tmpファイル → fsync → rename（部分書き込み防止） |
| **自動バックアップ** | 確定時バックアップ + 毎日3時の日次バックアップ（30日保持） |
| **入力バリデーション** | 全APIエンドポイントで型・範囲・整合性チェック |
| **XSS対策** | Flaskテンプレート自動エスケープ + フロント側escHtml() |
| **パストラバーサル対策** | ward/年月のホワイトリスト検証 |
| **CORS** | localhost限定 |
| **サブプロセス隔離** | OR-Toolsクラッシュがサーバーに波及しない |
